Technical Deep Dive

Platform Architecture

Click the tabs to explore each layer. Expand any accordion row for detail, code examples, and official links.

Databricks uses a two-plane architecture that separates orchestration from computation. The Control Plane is Databricks-managed. The Data Plane runs entirely inside your cloud account — your data never leaves your VPC.

All compute (Spark clusters) runs in the Data Plane on VMs you own. The Control Plane only sends orchestration signals — cluster commands, job triggers — never touching the data being processed.

Storage sits below compute in object storage (S3/ADLS/GCS), with Delta Lake adding ACID transactions, versioning, and schema management on top of Parquet files.

All-Purpose clusters are long-running, interactive clusters for notebook development and ad-hoc analysis. Multiple users can share one cluster. They are more expensive than Job clusters — always set auto-termination (30–60 min idle).

Job clusters are created when a job starts and terminated when it completes. Most cost-efficient for production: you only pay for the job duration. Each run is isolated — no state leaks between runs.

Delta Lake adds ACID transactions to object storage. If a write fails midway, the partial write is rolled back automatically. MERGE is the most powerful Delta command — upserts, deletes, and inserts in one atomic operation.

MERGE — atomic upsert
SQL
MERGE INTO silver.customers AS tgt
USING incoming AS src
  ON tgt.customer_id = src.customer_id
WHEN MATCHED THEN UPDATE SET
    tgt.email = src.email,
    tgt.updated_at = src.updated_at
WHEN NOT MATCHED THEN INSERT *
WHEN NOT MATCHED BY SOURCE THEN DELETE;
Delta updates & deletes →

Every write creates a new version in the transaction log. Query any historical version for auditing, reproducing ML experiments, or recovering from accidental deletes. Default retention is 30 days.

Time Travel queries
SQL
-- Query as of a specific date
SELECT * FROM silver.customers
TIMESTAMP AS OF '2024-01-15';

-- Query a specific version number
SELECT * FROM silver.customers
VERSION AS OF 5;

-- View full history
DESCRIBE HISTORY silver.customers;

-- Restore to previous version
RESTORE TABLE silver.customers
TO VERSION AS OF 3;
Delta time travel docs →

Streaming pipelines produce many small files that hurt read performance. OPTIMIZE compacts them. Z-ORDER BY co-locates related rows so filters on those columns can skip entire files (data skipping).

OPTIMIZE and VACUUM
SQL
-- Compact + co-locate by customer_id & date
OPTIMIZE silver.orders
ZORDER BY (customer_id, order_date);

-- Remove old files (keep 7 days for time travel)
VACUUM silver.orders RETAIN 168 HOURS;

-- Enable auto-optimize on write
ALTER TABLE silver.orders
SET TBLPROPERTIES (
  'delta.autoOptimize.optimizeWrite' = 'true',
  'delta.autoOptimize.autoCompact'   = 'true'
);

The Medallion Architecture organises data into three progressive quality layers. Data only flows forward. Each layer serves a different audience and purpose.

Bronze is the landing zone. No transformations, no cleaning. Use Auto Loader (cloudFiles) for incremental ingestion — it checkpoints which files have been processed. Add metadata columns: _ingested_at, _source_file.

Raw, unmodified dataAuto Loader (cloudFiles)Add ingestion metadataAppend-only writes

Silver applies data quality rules — remove duplicates, cast types, parse dates. Use DLT @expect decorators to declare quality constraints. Silver is the layer used by data scientists and SQL analysts.

DeduplicatedSchema enforcedDLT expectationsUsed by analysts/scientists

Gold tables are business-level aggregates — daily revenue, MAU, customer LTV. Wide, denormalised, optimised for fast BI queries. Power BI, Tableau, and Looker connect to Gold via SQL Warehouses or JDBC/ODBC.

Unity Catalog uses a three-level namespace: Catalog → Schema → Table. A metastore (one per region) contains multiple catalogs. Always use the full three-part path in production SQL: my_catalog.silver.orders.

Unity Catalog uses SQL GRANT/REVOKE. Row filters and column masks restrict what specific users see — mask PII for non-privileged users, filter rows by region.

Access control
SQL
-- Grant SELECT on a table to a group
GRANT SELECT ON TABLE gold.revenue
  TO `[email protected]`;

-- Grant usage on an entire schema
GRANT USAGE ON SCHEMA gold
  TO `data-team`;

-- Create a row access policy
CREATE ROW ACCESS POLICY region_filter
  AS (region STRING)
  RETURN region = current_user_region();
